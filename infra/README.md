# Virtual Waiting Room — CDK deployment

This CDK app (Python, `aws-cdk-lib` v2) deploys the Virtual Waiting Room:

- A `WaitingRoom` DynamoDB table (on-demand) with the `WaitingIndex` and
  `EligibilityIndex` GSIs and a stream (`NEW_AND_OLD_IMAGES`).
- Three Python 3.12 Lambda functions wiring `waiting_room.handlers`
  (`admit_handler`, `status_handler`, `promote_handler`).
- An HTTP API with `POST /admit` and `GET /status`.
- A 1-minute EventBridge rule that runs a promotion + expiry cycle for a demo
  event.
- A generated Secrets Manager secret holding the Entry_Token signing key; the
  admit/status Lambdas receive only its ARN and fetch the value at runtime.

The Lambda code asset is the repository root minus dev/test/build files, so
**no Docker is required** — the handlers depend only on the standard library
and the `boto3` already present in the Lambda runtime.

## Prerequisites

- Node.js (for the AWS CDK CLI): `npm install -g aws-cdk`
- Python 3.11+ and AWS credentials configured (`aws configure`)

## Deploy

```bash
cd infra
python -m venv .venv
# Windows: .venv\Scripts\activate
# macOS/Linux: source .venv/bin/activate
pip install -r requirements.txt

# One-time per account/region:
cdk bootstrap

cdk synth      # optional: inspect the synthesized template
cdk deploy
```

On success the stack outputs:

- `TableName` — the DynamoDB table name (`WaitingRoom`)
- `ApiUrl` — the HTTP API base URL (call `POST {ApiUrl}/admit`, `GET {ApiUrl}/status`)
- `EntryTokenSecretArn` — the ARN of the generated signing secret

## Seed a demo event

The scheduled promoter targets the `demo-event` id, but the event's `CONFIG`
and `CAPACITY` control items must exist first. Seed them with the project's
helpers (e.g. `waiting_room.provisioning.seed_event`) pointed at the deployed
table, or via the app once you have credentials.

## Teardown

```bash
cd infra
cdk destroy
```

The table and secret use `RemovalPolicy.DESTROY`, so teardown removes them
cleanly (all queued data is deleted).

## Notes

- **Status aggregates simplification.** `status_handler` computes queue
  position from the sparse `WaitingIndex` (bounded `Query` calls, never a
  `Scan`) and reads `Promoted_Total` from the `CAPACITY` item. The promotion
  rate used for `Estimated_Wait` is a fixed positive constant, not an observed
  rate — an intentional approximation that keeps the deployment free of
  ElastiCache.
- **Secret handling.** The signing key is generated by Secrets Manager and read
  at runtime by ARN, so it never appears in a plaintext Lambda environment
  variable.
