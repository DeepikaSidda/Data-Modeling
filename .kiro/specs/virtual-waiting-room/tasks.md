# Implementation Plan: Virtual Waiting Room

## Overview

This plan implements the Virtual Waiting Room in **Python 3.11+**, built bottom-up and test-driven. It uses `boto3` for DynamoDB access, **Hypothesis** for property-based tests (each property test runs **≥ 100 iterations**), `pytest` for unit/integration tests, and **moto** / **DynamoDB Local** for integration tests.

All source code lives under a new `waiting_room/` package and all tests under `tests/`. The frozen deliverables under `submission/` (`nosql-workbench-model.json`, `design-document.md`, `access-pattern-matrix.md`) are **never** modified or regenerated by any task.

The build order is: (1) scaffolding + table provisioning, (2) pure-logic layer with property-based tests referencing the design's Correctness Properties, (3) the DynamoDB data-access layer wiring pure logic to the table, (4) integration tests against DynamoDB Local / moto, (5) a load/burst smoke harness. Property tests are placed next to the logic they validate to catch errors early. Each property test is tagged in the format `Feature: virtual-waiting-room, Property {number}: {property_text}`.

  - [x] 2.1 Implement the Ordering_Key allocator and comparator
    - Implement `waiting_room/ordering.py` producing `<seq>#<tiebreak>` where `seq` is a Hybrid Logical Clock value (48-bit physical millis + 16-bit logical counter) rendered as a fixed-width, zero-padded, lexicographically sortable string, and `tiebreak` is a server CSPRNG token (e.g., 64-bit hex from `secrets`)
    - Guarantee the HLC advances monotonically per node even when the wall clock regresses or skews beyond the configured bound
    - Provide a comparator function that yields a strict total order over Ordering_Keys
    - _Requirements: 3.1, 3.3, 3.4, 3.6, 3.7_

  - [x]* 2.2 Write property test for HLC monotonicity under clock skew
    - **Feature: virtual-waiting-room, Property 7: For any stream of admissions processed by a node — including injected clock regressions and cross-node skew beyond the configured bound — the emitted seq component of the Ordering_Key is monotonically non-decreasing in admission order for that node.**
    - **Validates: Requirements 3.1, 3.6**
    - ≥ 100 iterations with generators injecting clock regressions and same-millisecond clusters

  - [x]* 2.3 Write property test for total, deterministic ordering and idempotent position
    - **Feature: virtual-waiting-room, Property 8: For any set of Queue_Entries, the Ordering_Key comparator is a strict total order (irreflexive, antisymmetric, transitive, and total — no two distinct entries compare equal because of the unique random tie-breaker), and recomputing Queue_Position over the same set any number of times yields identical results.**

## Notes

- Tasks marked with `*` are optional (tests, load harness, and AWS-only deployment tasks) and can be skipped for a faster local MVP; core implementation tasks are never optional.
- Every property test references a specific design Correctness Property using the tag format `Feature: virtual-waiting-room, Property {number}: {property_text}` and runs with **≥ 100 iterations** via Hypothesis.
- Concurrency properties (11, 12, 13) use a deterministic scheduler / interleaving explorer over a modeled atomic-counter and conditional-write store so races are reproducible.
- Integration tests run against DynamoDB Local / moto; property tests run against the pure-logic layer with DynamoDB mocked.
- No task modifies, regenerates, or touches anything under `submission/` — those three deliverables are frozen.
- Each task references the specific requirements it implements for traceability.
